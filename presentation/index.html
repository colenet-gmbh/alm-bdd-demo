Was ist BDD?

"Behavior Driven Development"

Spezifizierung des Verhaltens einer Software in textueller Form

Testing des spezifizierten Verhaltens mit Fallbeispielen

Einbeziehung von Stakeholder in die Anforderungsanalyse

(M) BDD => "Behavior Driven Development" 1. Schritt: Verhalten verbindlich dokumentieren (textuell!) 2. Schritt: Verhalten mit Fallbeispielen testen Ermöglicht Einbeziehung von Erwartungen und Anforderungen von Stakeholdern → Stakeholder hat Einfluss auf Verhalten der Software

Wie hilft uns BDD agiler zu werden?

Stärkt Zusammenarbeit innerhalb des Teams und mit den Stakeholdern

Stärkt das gemeinsame Verständnis von Software

Ermöglicht schnellere Feedback Loops

(H) Stärkt Zusammenarbeit → Gemeinsames Verständnis → Schnelleres Feedback, da direkte Kommunikation

Wie funktioniert BDD?

(M) 1. BDD ist eine Weiterentwicklung von TDD 2. Zwei Zyklen. BDD & TDD Cycle. 3. BDD lässt sich runterbrechen auf "Discover, Formulate, Automate" 4. Erklären welche Schritte PO & Dev zusammen durchführen

Feature Files

                Feature: Rescue operation

    # Ein Szenario beschreibt ein Verhalten
    Scenario: Firefighters saves a kitten
        # Setzt eine Situation oder einen State voraus

        # Spezifiziert eine Aktion

(H) Beispiel eines features files - Feature: NAME des Features, Platz darunter für nähere Beschreibung - Scenario => Verhalten - Given => Voraussetzung - WHEN => Aktion - THEN => Resultat/Ergebnis

BDD Anti-Patterns

In der Arbeitsweise mit BDD

Verhalten wird definiert, nachdem Code implementiert wurde

Product Owner schreibt Szenarien im Alleingang

Szenarien enthalten zu viele Informationen

(H) - Verhalten wird nachträglich dokumentiert → Annahmen bei der Implementierung sind nicht nachvollziehbar; Wissen liegt bei einer Person - Der PO zieht das Dev Team für die Definition der Szenarien nicht hinzu → Oft muss nachgehakt; Randfälle etc.

Gherkin Best-Practices

(M) 1. Gherkin ist eine Feature-Definition Language 2. Grammatikalische Regeln die ein Feature für Cucumber definieren. 3. In der Arbeit mit Gherkin kann man viel falsch machen. 4. In dieser Sektion wollen wir Hilfestellungen geben.

Gherkin Best-Practice #1 - Die goldene Regel

Lesbarkeit und Verständnis steht über Allem

                Feature: Fire department

  Scenario: Forest fire
    Given A forest has caught fire
    When The wildfire is detected
    And The fire department extinguishes the wildfire
    Then State of the forest fire is "extinguished"
(M) 1. Lesbarkeit ist in Gherkin genauso wichtig wie bei Code. 2. Damit Code gelesen und angepasst werden kann er verständlich sein. 3. ... Beispiel erläutern 4. Goldene Regel, weil die Hilfestellung von BDD ohne Lesbarkeit dem Team keinen Vorteil bringt. 5. Eher Nachteil. Teams können kein gutes Verhalten entwickeln und sind verwirrt.

Gherkin Best-Practice #2 - One-to-one Regel

Jedes Szenario beschreibt genau ein Verhalten

1 Verhalten -> 1 Beispiel -> 1 Szenario -> 1 Test -> 1 Resultat

                Feature: Fire department

  Scenario: Emergency call
    Given An emergency
    When Emergency call received by emergency hotline
    Then Fire departments state is "alert"

  Scenario: Fire fighting
    Given A car caught fire
    When Fire truck extinguishes burning car
    Then State of the car is "extinguished"
(M) 1. One-to-one: Jedes Szenario beschreibt genau ein Verhalten. 2. Ein Verhalten produziert ein messbares Resultat 3. ... Beispiele erläutern

Vorteile der One-To-One Regel...

Jeder Testfehler zeigt auf ein einzigartiges Problem

Weniger komplexe Arbeit ermöglicht schnellere Feedback Loops

Erhöht die Lesbarkeit

Gherkin Best-Practice #3 - Grammatik

Verwenden der korrekten Grammatik hilft dabei das Verständnis zu erhöhen

                Feature: Fire department

  Scenario: Fire fighting
    # Vermeidung der Vermischung von verschiedenen Erzählformen
    # (entweder Erste oder Dritte Person)
    # Given -> Past or present-perfect tense
    Given A car that has caught fire
    # When -> Present tense
    When Fire truck extinguishes burning car
    # Then -> Present or future tense
    Then State of the car is "extinguished"
(M) 1. Die richtige Grammatik hilft die Lesbarkeit zu steigern und zahlt auf die goldene Regel ein. 2. Keep it simple! Versuchen im Present / Präsens zu bleiben. 3. ... Beispiele erläutern 4. Was wäre wenn perfect tense, future tense etc.

Gherkin Anti-Patterns

(M) 1. Genauso wie Best Practices gibt es Anti-Patterns für Gherkin. 2. Davon wollen wir euch auch ein paar vorstellen.

Gherkin Anti-Pattern #1

Verschachtelte When-Then-When-Then Szenarien

                Feature: Fire department

  Scenario: Fire fighting
    Given A car that has caught fire
    When Emergency call received
    Then Department receives alarm
    When fire truck leaves fire department
    Then fire truck arrives at the place of the accident
    When fire truck extinguishes the burning car
    Then State of the car is "extinguished"
(M) 1. ... Beispiele erläutern 2. Verstößt gegen die One-to-one Regel. 3. Reduziert die Lesbarkeit und verstößt gegen die goldene Regel. 4. Erhöht die Komplexität des Softwaretests durch viele Bedingungen und Assertions.

Gherkin Anti-Pattern #2

Beschreibung der Aktionen anstatt Beschreibung des Verhaltens

                Feature: Fire department

  Scenario: Rescue operation
    Given A man is trapped in his car after an accident
    When 3 Fire trucks with 4 firefighters each arrive
      And Firefighters secure the place of the accident
      And Firefighters ...
      And ...
      ...
    Then State of the accident participant is "rescued"
(M) 1. ... Beispiele erläutern 2. Aktionen beinhalten Details; Implementierung der Software wird fest definiert. 3. BDD spezifiziert das Verhalten, um die Implementierung offenzulassen. 3. Das Feature File muss in Zukunft geändert werden. 4. Feature Files sollten nur geändert werden wenn sich das Verhalten ändert.

(H) Zunächst römische Ziffer zu Dezimalzahl Später in einer Livedemo Implementierung von römischer Ziffer zu Dezimalzahl

Erstes Featurefile

Feature: Roman Numerals

  Converts roman numerals to decimal numbers

  Scenario:
    Given The roman numeral "II"
    When Converted to decimal number
    Then The output should equal 2
(H) SELBSTERKLÄREND

Tests

    @Given("The roman numeral {string}")
    fun `Given the roman numeral`(roman: String) {
        romanNumber = roman
    }

    @When("Converted to decimal number")
    fun `When converted to decimal number`() {
        decimalNumber = numeralsConverter.convert(romanNumber)
    }

    @Then("The output should equal {int}")
    fun `Then the output should equal`(exp: Int) {
        assertNotNull(decimalNumber)
        assertEquals(exp, decimalNumber)
    }
(H) Implementierung der Tests kommt zuerst, nach Prinzip TDD. - gegeben String, class field wird gesetzt - converter interface wird definiert; methodenaufruf; rückgabewert als class field - test ob vorheriger rückgabewert dem erwarteten wert entspricht

Implementierung

class NumeralsConverter {

    private val romansToDecimals = mapOf(
        "II" to 2
    )

    fun convert(romanNumeral: String): Int? = romansToDecimals[romanNumeral]
}
(H) Sehr heruntergebrochene implementierung um tests grün zu bekommen → Scenario erfüllen

(H) Jetzt kommt der, für uns Entwickler wohl interessante Teil. Wir möchten mithilfe des Jira Plugins ein neues Feature ermöglichen. Teil der Einrichtung ist die Git Pipeline, wir erinnern uns (Automatisiert und nachvollziehbar). Dazu wird Moritz euch unsere Github Pipeline näherbringen, um zu zeigen, wie die Integration bei uns erfolgt ist.

CI Pipeline

            name: Cucumber Test

on: [push, pull_request]

jobs:
  bdd_test:
    runs-on: ubuntu-latest
    name: BDD Test
    steps:
    - uses: actions/checkout@v3
    - uses: actions/setup-java@v3
      with:
        java-version: 8
        distribution: zulu
    - name: Run Cucumber tests
      run: |
        ./gradlew test
    - name: Upload Cucumber test results
      if: always()
      run: |
        curl -X POST https://c4j.cucumber.io/ci/rest/api/results \
          -H 'authorization: Bearer ${{ secrets.CUCUMBER_FOR_JIRA_CREDS }}' \
          -H 'content-type: multipart/form-data' \
          -F 'results_files[]'=@build/cucumber-reports/Cucumber.json \
          -F language=jvm
(M) 1. GitHub Actions Pipeline 2. Läuft auf Pull-Requests und bei jedem Push auf einen Branch 3. Tests laufen lassen 4. Tests hochladen
